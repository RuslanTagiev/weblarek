# Интернет-магазин «Web-Larёk»
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

### Данные и модели данных (Model)

#### Данные

Интерфейс товара: 
`(IProduct)` - Описывает структуру данных отдельного товара:
`id: string` — уникальный идентификатор;
`description: string` — описание;
`image: string` — ссылка на изображение;
`title: string` — название;
`category: string` — категория;
`price: number | null` — цена в синапсах.

Интерфейс покупателя: 
`(IBuyer)` - Описывает данные, вводимые при оформлении заказа.
`payment: string` — вид оплаты;
`email: string` — почта;
`phone: string` — телефон;
`address: string` — адрес доставки.

### Модели данных:

#### Класс CatalogModel
  Отвечает за хранение массива товаров и управление предпросмотром.

Поля:
`_items: IProduct[]` — массив товаров.
`_preview: IProduct | null` — выбранный товар.

Методы:
`setItems(items: IProduct[])` — сохранить список товаров.
`getItems()` — получить список товаров.
`getItem(id: string)` — найти товар по id.
`setPreview(item: IProduct)` — выбрать товар для модалки.
`getPreview()` — получить выбранный товар.

#### Класс BasketModel 
 Управляет товарами в корзине покупателя.

Поля:
`_items: IProduct[]` — товары в корзине.

Методы:
`addItem(item: IProduct)` — добавить товар.
`removeItem(id: string)` — удалить товар.
`clear()`— очистить корзину.
`getTotal()` — получить общую стоимость.
`getCount()` — получить количество товаров.
`isInBasket(id: string)` — проверить наличие товара в корзине.

#### Класс BuyerModel 
Хранит данные покупателя и отвечает за их валидацию.

Поля: 
`payment: string` — выбранный способ оплаты (например, «онлайн» или «при получении»);
`address: string` — полный адрес доставки, введенный пользователем;
`email: string` — адрес электронной почты для отправки чека и подтверждения заказа;
`phone: string` — контактный номер телефона для связи службы доставки с клиентом;
`formErrors: FormErrors` — объект с описанием ошибок валидации (где ключ — название поля, а значение — текст ошибки).


Методы:
`setData(field: keyof IBuyer, value: string)` — сохранение значения конкретного поля.
`getData()` — получение всех данных покупателя.
`clear()` — очистка данных.
`validate()` — проверка данных на заполненность. Возвращает объект с текстом ошибок для каждого поля или пустой объект, если всё валидно.


### Слой коммуникации.

#### Класс LarekApi 
 Принимает участие в обмене данными между сервером и приложением. Расширяет базовый класс Api и реализует интерфейс для работы с конкретными эндпоинтами магазина «Веб-ларек».

Зона ответственности:
-Получение списка товаров с сервера и приведение их к типизированному массиву.
-Отправка данных заказа на сервер и получение результата оформления.
-Формирование полных ссылок на изображения товаров путем объединения пути из данных товара и базового URL (CDN).

Конструктор:
`constructor(cdn: string, baseUrl: string, options?: RequestInit)` — принимает базовый URL для контента (изображений), базовый URL API и объект с настройками запроса (заголовки).

Поля:
`cdn: string` — базовый путь к изображениям.

Методы:
`getProductList(): Promise<IProductList>` — получает объект со списком товаров с сервера. Полученный массив items используется для наполнения модели данных каталога.
`orderProducts(order: IOrder): Promise<IOrderResult>` — передает данные заказа (список ID товаров и данные покупателя) на сервер методом post. Возвращает объект с ID заказа и итоговой суммой.


### Слой представления (View)

Все классы представления наследуются от базового класса `Component`. Они отвечают за отображение интерфейса и генерацию событий при взаимодействии пользователя с элементами.

### Класс Modal 
Универсальный контейнер для модальных окон. Управляет отображением контента, блокировкой скролла и закрытием.
 Поля: 
 `content` — HTMLElement для вставки в модальное окно .

 ### Класс Form
 Базовый класс для всех форм проекта. Управляет валидацией отображением ошибок и активностью кнопки сабмита.
 
 Сеттеры:
 `valid` — активность кнопки
 `errors` — текст ошибки

#### Класс Card (и его наследники)
Базовый класс для отображения товара.
 `CardCatalog` — для отображения в галерее на главной.
 `CardPreview` — детальное описание товара в модальном окне с кнопкой добавления/удаления.
 `CardBasket` — компактная строка в корзине с индексом и ценой.

#### Класс Basket
Отвечает за отображение списка выбранных товаров, расчет общей стоимости и переход к оформлению заказа.

#### Класс Page
Управляет основными элементами страницы: галереей товаров и счетчиком на иконке корзины в шапке.



### Список событий в приложении

#### События пользовательского интерфейса (View -> Presenter)
 `card:select` — выбор карточки в каталоге для открытия окна предпросмотра.
 `card:toBasket` — нажатие кнопки «В корзину» или «Удалить» в окне предпросмотра.
 `basket:open` — клик по иконке корзины в шапке сайта.
 `basket:delete` — нажатие на иконку удаления в строке товара внутри корзины.
 `order:open` — нажатие кнопки «Оформить» в корзине.
 `order:submit` — сохранение данных первого шага оформления и переход к форме контактов.
 `contacts:submit` — финальная отправка данных заказа на сервер.
 `modal:open` — уведомление об открытии модального окна (используется для блокировки прокрутки страницы).
 `modal:close` — уведомление о закрытии модального окна (используется для разблокировки прокрутки).

#### События изменения данных (Model -> Presenter)
 `items:changed` — каталог товаров загружен или изменен (требуется перерисовка галереи).
 `basket:changed` — состав корзины изменился (требуется обновить счетчик в шапке и список товаров в корзине).
 `formErrors:change` — состояние валидации форм изменилось (требуется обновить отображение ошибок в интерфейсе).


 ## Взаимодействие слоев (Презентер)

В данном проекте роль Презентера выполняет основной скрипт `main.ts`, который связывает слои Model и View через брокер событий.

**Логика взаимодействия:**
1 - Инициализация: В презентере создаются экземпляры всех моделей данных и компонентов представления, а также инициализируется API.
2 - Подписка: Презентер устанавливает слушатели на события, генерируемые моделями и представлениями.
3 - Реакция на изменения:
   - При действиях пользователя (клик, ввод данных) View генерирует событие. Презентер вызывает методы Модели для обновления данных.
   - Когда данные в Модели меняются, она генерирует событие об обновлении. Презентер ловит его и передает свежие данные в методы `render` соответствующих компонентов View.
